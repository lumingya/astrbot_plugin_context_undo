# astrbot_plugin_context_undo

为每次成功的 AI 回复建立一个“编号快照”。

撤回时不再从当前 `history` 尾部猜测要删多少条，而是直接恢复到对应编号之前的快照，同时同步恢复 LTM，并清理易失的会话状态。

## 用法

- `撤回` / `回滚`
  - 回滚最新一条已编号上下文
- `撤回列表` / `回滚列表` / `撤回 list`
  - 查看当前对话最近的编号
- `撤回 #3`
  - 回滚到 `#3` 之前，会同时移除 `#3` 以及之后的所有编号
- `撤回 最近 2`
  - 一次回滚最近 2 条已编号上下文
- `/undoctx`
- `/undoctx list`
- `/undoctx 3`
- `/undoctx last 2`
- 自然语言触发只接受以上这些明确格式，不会再按“包含撤回/回滚关键词”误判普通句子

## 行为

- 在每次成功的 LLM 回复前记录当前会话 `history` 和 LTM 快照
- 回复成功后为这次对话分配递增编号
- 编号快照按 `conversation_id` 持久化保存，不再只挂在当前会话指针上
- 编号快照不再按固定时长自动过期；会按每个会话最多保留最近 `48` 条
- 群聊 LTM 的实时 `session_chats` 也会按 `conversation_id` 单独持久化，并在重启或切换对话后自动恢复
- 回滚时直接恢复到对应编号之前的快照
- 仅管理员可以使用撤回相关命令；非管理员会收到 warning
- 当前会话被 `reset` / `del` / `delete` 时，会同时清空该会话的编号快照和实时 LTM 状态
- history 清空时，该会话的编号会重新从 `#1` 开始
- 切换到别的对话或重启 AstrBot 时，旧对话的编号快照会保留；切回原对话后仍可继续使用

## 为什么这样更稳

旧实现是“从当前 history 尾部裁剪 N 轮”。一旦中间混入 tool 记录、审核文本、LTM 或其他会话状态，就可能出现“提示撤回成功，但后续回答还受旧内容影响”。

新版改为“按编号恢复检查点”，目标更明确，也更容易核对。
